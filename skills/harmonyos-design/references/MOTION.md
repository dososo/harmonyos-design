# 动效、手势与异步状态

> 状态：V0.2 草案  
> 主要来源：OpenHarmony UX 动效规范、ArkUI 动画衔接与曲线文档  
> 最后核验：2026-07-11

## 1. 动效的职责

动效只应承担以下一种或多种任务：

- 立即反馈输入；
- 解释状态变化；
- 表达同层、上下层或跨层关系；
- 维持对象身份和空间连续性；
- 引导注意力；
- 降低突变和等待焦虑；
- 在少数合适时刻表达品牌气质。

没有明确职责的动效，优先删除。

## 2. 动效不能替业务状态“撒谎”

这是项目不可妥协的 House Rule，针对后端延迟造成的虚假成功问题，也符合状态反馈必须真实的基本交互原则。

必须区分：

```text
idle
→ acknowledged：本地已收到输入
→ pending：远端或耗时任务处理中
→ confirmed / failed：真实结果已确认
```

### 规则

1. 按下反馈只表示“系统听到了”，不表示远端已成功。
2. 成功勾选、完成转场和成功播报只在真实状态确认后触发。
3. 动画时长不得作为后端完成计时器。
4. 乐观更新必须有可靠回滚，且不能让用户误以为不可逆结果已确认。
5. 支付、删除、权限、发布和跨设备同步默认采用保守确认。
6. 处理状态必须有超时、失败、重试、取消和重复提交策略。
7. 响应乱序时只允许最新有效请求改变最终状态。

### 错误示例

```text
点击“支付”
→ 立即显示成功勾选
→ 后端仍在请求
→ 数秒后失败并突然回滚
```

### 推荐状态编排

```text
按下：即时局部反馈
提交：进入明确 pending
等待：保持状态真实，必要时允许取消
成功：确认后播放完成反馈
失败：解释原因，恢复可操作状态
```

## 3. 交互的三个阶段

### 3.1 按下

OpenHarmony 公开手势动效资料指出，点击由按下和抬起组成，按下接触平均会持续一段时间，因此反馈应在按下时开始，而不是抬起后才出现。

检查：

- 按下是否立即可见；
- 抬起是否提交；
- 移出目标后能否取消；
- 回到目标后是否恢复；
- 系统 Button 是否已提供默认反馈。

自定义缩放值属于 House Style。不要把 `0.97` 写成 HarmonyOS 官方规定。

### 3.2 跟手

拖拽、滑动、捏合和翻动在跟手阶段应持续响应。

优先：

```ts
curves.responsiveSpringMotion()
```

其公开 ArkUI 默认参数为：

| 参数 | 默认值 | 类型 |
| --- | ---: | --- |
| response | 0.15s | ARKUI_DEFAULT |
| dampingFraction | 0.86 | ARKUI_DEFAULT |
| overlapDuration | 0.25s | ARKUI_DEFAULT |

这些默认值来自 API 文档，不是所有自定义动效的统一设计 Token。

### 3.3 离手与回稳

离手后应从当前显示状态继续，必要时继承速度。

优先：

```ts
curves.springMotion()
```

公开 ArkUI 默认参数：

| 参数 | 默认值 | 类型 |
| --- | ---: | --- |
| response | 0.55s | ARKUI_DEFAULT |
| dampingFraction | 0.825 | ARKUI_DEFAULT |
| overlapDuration | 0s | ARKUI_DEFAULT |

当同一对象同一属性的弹性动画重新定向时，`springMotion` 会替换前一动画并继承速度。

## 4. 可打断与动画衔接

用户意图可以在动画过程中改变。

必须检查：

- 新输入能否立刻接管；
- 动画是否从当前显示状态继续；
- 目标反向时是否有跳变；
- 输入是否被转场锁住；
- 速度是否突然归零；
- 连续点击是否产生排队动画。

ArkUI 的动画衔接能力允许在目标变化时平滑过渡。跟手阶段使用 `responsiveSpringMotion`、离手阶段使用 `springMotion` 是公开文档给出的典型组合。

## 5. 弹性曲线选择

### 使用弹性曲线

- 连续手势；
- 拖拽释放；
- 目标不断变化；
- 需要继承速度；
- 回到边界或吸附点；
- 对象具有明确物理语义。

### 优先非弹性曲线

- 简单颜色变化；
- 非手势的小型状态淡入淡出；
- 不需要速度继承的短提示；
- 明确、克制的进入和退出。

不要只因“更灵动”加入弹跳。大面积、多次振荡会放大干扰。

## 6. 弹簧参数

### `springMotion`

设计师友好的参数：

- `response`：无阻尼自然振荡周期，影响复位速度；
- `dampingFraction`：阻尼比；
- `overlapDuration`：两段弹性动画衔接时参数过渡时间。

### `interpolatingSpring`

显式参数：

- velocity；
- mass；
- stiffness；
- damping。

其 velocity 是归一化速度：实际速度除以动画属性改变量。

### 重要 Gotcha

对 `springMotion`、`responsiveSpringMotion` 和 `interpolatingSpring`，实际时长由物理参数、速度和算法决定。公开文档说明相关 `duration` 不控制这类物理曲线的实际持续时间。

因此：

- 不用动画结束时间驱动业务；
- 不用固定延迟假装等待弹簧完成；
- 业务状态和视觉状态分离。

## 7. 非弹性曲线语义

OpenHarmony 公开动效资料给出的主要语义：

### 标准曲线

```text
cubic-bezier(0.40, 0.00, 0.20, 1.00)
```

适用：对象在运动前后始终处于视线内，例如 Tab、Switch、缩放或布局内状态变化。

### 减速曲线

```text
cubic-bezier(0.00, 0.00, 0.40, 1.00)
```

适用：从视线外或不可见状态进入，并在终点稳定。

### 加速曲线

适用：对象从当前视线中退出或被移除。

### ArkUI 曲线枚举

公开 API 包括：

- `Curve.FastOutSlowIn`；
- `Curve.LinearOutSlowIn`；
- `Curve.FastOutLinearIn`；
- `Curve.ExtremeDeceleration`；
- `Curve.Sharp`；
- `Curve.Rhythm`；
- `Curve.Smooth`；
- `Curve.Friction`。

不要凭名字机械选择；以视觉语义和目标 SDK 文档为准。

## 9. 转场和层级

### 同层级

用于 Tab、编辑模式、横竖屏等平级状态。运动不应误导为进入子页面。

### 上下层级

用于列表到详情、新建、搜索、解锁等父子关系。进入和返回应保持方向、来源和对象身份。

### 跨层级

用于应用间或任务间跳转。不要用局部微动效掩盖上下文切换。

## 10. 一镜到底与共享关系

公开规范推荐利用：

- 共享元素；
- 共享容器；
- 共享动势。

ArkUI 可使用：

```ts
.geometryTransition('sharedId')
```

检查：

- 两端 ID 是否一致；
- 默认 Navigation 转场是否需要关闭；
- 共享对象是否真的是用户关注焦点；
- 容器大小、圆角和内容切换是否协调；
- 中断和返回是否保持身份。

不要为了炫技把所有页面都做成共享转场。

## 11. 边界与阻尼

边界处逐渐增加阻力，比硬停止更能保持响应感。

适用：

- 列表过界；
- 卡片横滑；
- 下拉返回；
- 面板拖动；
- 翻页提示。

检查：

- 过界程度是否可控；
- 是否暗示存在更多内容；
- 回弹是否继承速度；
- 是否与系统滚动行为冲突；
- 多输入环境是否仍合理。

具体阻尼函数属于实现策略，不要把其他平台的公式标成鸿蒙官方。

## 12. 动效性格与反同质化

动效必须匹配产品：

| 产品 | 倾向 |
| --- | --- |
| 金融/支付 | 可信、克制、状态真实 |
| 生产力工具 | 快、低干扰、键盘友好 |
| 内容消费 | 连续、空间清晰 |
| 儿童/教育 | 可解释、适度愉悦 |
| 车机/大屏 | 远距清晰、低认知负担 |
| 穿戴 | 短、明确、省电 |

禁止把所有产品统一成：

- 模糊玻璃；
- 大圆角卡片；
- 弹性入场；
- 错峰瀑布；
- 发光和拖尾；
- 某类流行的统一视觉风格。

审视时明确：

```text
应保留的产品特征：
不应套用的通用风格：
允许的 House Style：
```

## 13. 错峰与微动效

错峰入场、数值脉冲、图标微动效属于可选 House Style。

使用条件：

- 低频；
- 不阻止交互；
- 不延迟信息读取；
- 能说明顺序或变化；
- 大字体和低运动状态下可替代。

不要给每个列表刷新都加错峰，也不要在数值频繁变化时持续脉冲。

## 14. 低运动替代

当前公开来源不足以在本项目中指定一个跨版本统一的 HarmonyOS“减少动态效果”API，因此不要发明 API 名称。

设计层必须准备：

- 位移动画改为短透明度变化；
- 去除过界、弹跳和视差；
- 保留必要状态反馈；
- 成功、错误和加载语义不依赖运动；
- 大面积移动改为局部变化；
- 在目标 SDK 和设备上核验系统偏好读取能力。

该要求在规则库中标为 H4，直到获得目标 HarmonyOS 版本的 H1/H2 证据。

## 15. 性能

优先：

- 系统动画接口；
- `transition` 处理出现和消失；
- 图形变换处理连续位置和大小变化；
- 同参数属性合并到同一 `animateTo`；
- 减少冗余状态刷新。

避免：

- 手写 `setTimeout` 逐帧动画；
- 高频修改 `width` / `height` 触发布局；
- 多个相同参数的 `animateTo` 分散调用；
- 在手势高频回调中做耗时任务；
- 把模糊、阴影和复杂绘制当成免费效果。
