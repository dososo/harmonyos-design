# 无障碍与包容性设计

> 状态：V0.2 草案  
> 主要来源：OpenHarmony ArkUI 无障碍文档与 UX 自检表  
> 最后核验：2026-07-11

## 1. 基本立场

无障碍不是发布前补标签，而是组件语义、焦点、状态、布局和反馈的组成部分。

审视至少覆盖：

- 视觉障碍；
- 听觉障碍；
- 运动能力差异；
- 认知负担；
- 大字体；
- 强光、单手、远距等临时情境。

## 2. 可交互组件的三项基础

公开 ArkUI 文档指出，可交互组件要让辅助工具：

1. 能识别该组件；
2. 理解功能和操作信息；
3. 获得真实状态与行为。

对应能力包括：

- `accessibilityLevel`；
- `accessibilityText`；
- `accessibilityDescription`；
- `accessibilityRole`；
- `accessibilityChecked`；
- `accessibilitySelected`。

## 3. `accessibilityText`

用于为无文本内容提供简洁功能名称。

推荐：

```ts
Image($r('app.media.play'))
  .accessibilityText('播放')
```

要求：

- 简洁；
- 说明功能，不重复长篇操作指导；
- 与当前状态一致；
- 使用资源字符串支持本地化。

错误：

- “图片”；
- 文件名；
- 图标 Unicode；
- 冗长的“单指双击即可……”全部写入 text；
- 视觉已变为暂停但仍播报播放。

## 4. `accessibilityDescription`

用于补充：

- 后果；
- 当前不可用原因；
- 系统默认提示无法表达的信息；
- 高风险操作说明。

不要重复 label，也不要把所有帮助文档塞入 description。

## 5. Role 与状态

### `accessibilityChecked`

适用：

- Checkbox；
- Toggle；
- 二态或三态开关。

表达控件物理勾选状态。

### `accessibilitySelected`

适用：

- Tabs；
- 单选列表；
- 导航当前项。

表达互斥选择中的当前项。

不要互换二者。状态改变时，语义必须同步。

## 6. 分组

`accessibilityGroup(true)` 可把相关子节点作为整体，减少冗余播报。

适用：

- 日期；
- 卡片摘要；
- 图标 + 标题 + 描述构成一个操作；
- 组合状态。

避免：

- 把多个独立按钮分成一个无法操作的整体；
- 分组后丢失子项行为；
- 视觉顺序和播报顺序不一致。

## 7. 重要性

`accessibilityLevel` 控制组件及子组件是否被辅助服务识别。

公开值包括：

- `auto`；
- `yes`；
- `no`；
- `no-hide-descendants`。

使用原则：

- 关键操作可识别；
- 装饰内容不制造噪声；
- 不用 `no-hide-descendants` 隐藏仍需操作的子节点；
- 自定义组件需要真机验证辅助树。

## 8. 自绘组件

使用 `accessibilityVirtualNode` 为自绘图表、Canvas 或复杂视觉组件建立可访问语义。

必须考虑：

- 节点数量；
- 阅读顺序；
- 数据摘要；
- 选择和操作；
- 缩放或筛选后状态更新。

不要让图表只能通过颜色和位置理解。

## 9. 焦点

在键盘、遥控器、旋钮和辅助工具场景检查：

- 所有核心功能可到达；
- 焦点顺序符合任务；
- 当前焦点视觉清晰；
- 焦点不会被装饰动画抢走；
- Dialog 打开后焦点进入合理位置；
- 关闭后焦点返回触发源；
- 分栏和折行后顺序仍正确；
- 隐藏元素不保留焦点。

焦点态和激活态不能只用同一种视觉表现。

## 10. 大字体

OpenHarmony 公开自检要求字体使用 `fp` 并响应系统字体大小。

检查：

- 标题、正文、按钮和输入框；
- 文本是否裁剪；
- 按钮文字是否换行或溢出；
- 固定高度是否失效；
- Tab 标签是否可理解；
- Dialog 是否可滚动；
- 横竖屏和分屏；
- 中文、英文、数字和长本地化文本。

不要通过禁止字号缩放解决布局问题。

## 11. 色彩与状态

颜色不能成为唯一信号。

状态还应通过：

- 文本；
- 图标；
- 形状；
- 位置；
- 边框或图案；
- 无障碍状态；
- 必要的触觉或声音。

检查浅色、深色、高对比和色觉差异。

## 12. 动效与无障碍

项目不得发明未经核验的跨版本“减少动态效果” API。

设计层准备：

- 位移改短淡化；
- 去除弹跳、视差、过界；
- 保留状态和因果反馈；
- 大面积运动改局部；
- 不用运动表达唯一信息；
- 在目标 SDK 验证系统偏好读取能力。

### 异步状态播报

- 本地按下不播报“成功”；
- pending 应有真实处理语义；
- 远端确认后才播报完成；
- 失败和重试必须可理解；
- 防止重复请求导致重复播报。

## 13. 文案

- 按钮使用直接动作词；
- 中文按钮通常保持简洁；
- 不用“确定”替代具体行为；
- 危险操作说明后果；
- 错误提示说明发生了什么和下一步；
- 占位文本不替代 label；
- 图标按钮必须有可访问名称。

## 14. 测试流程

### 静态检查

- 非文本可点击组件；
- 状态属性；
- 资源字符串；
- 固定高度和字体；
- 隐藏与焦点；
- 自绘组件。

### 真机检查

1. 开启屏幕朗读；
2. 依次浏览关键路径；
3. 检查角色、名称、状态和提示；
4. 执行成功、失败、加载和取消；
5. 检查焦点返回；
6. 放大字体；
7. 切换方向和窗口；
8. 使用键盘或遥控器（适用时）。

## 15. Blocker

以下通常阻断通过：

- 关键按钮无法被识别；
- 图标按钮无功能名称；
- 成功/失败状态播报错误；
- 焦点陷阱；
- 大字体使核心任务不可完成；
- 自绘关键数据完全不可访问；
- 颜色是唯一的危险或错误信号；
- Dialog 关闭后焦点丢失且无法继续任务。
