# 设计原则

> 状态：V0.2 草案  
> 来源范围：公开 HarmonyOS 设计入口、OpenHarmony UX 与 ArkUI 文档  
> 最后核验：2026-07-11

## 1. 核心命题

```text
一致而不相同
连续而不阻塞
反馈即时但状态真实
基础原则持久，平台表现版本化
系统能力优先
证据可追溯
```



## 1.1 三层知识架构

### Evergreen Foundation

跨版本优先保持：

- 反馈与因果；
- 直接操控；
- 连续性和可打断；
- 速度衔接；
- 层级与方向；
- 可读性和无障碍；
- 用户控制与状态真实性。

### Versioned Platform Layer

随目标 HarmonyOS/OpenHarmony/ArkUI 版本核验：

- 当前视觉语言；
- 系统组件默认；
- API 和曲线；
- 资源 Token；
- 设备与输入能力；
- 性能工具。

### Project Overlay

由项目定义：

- 品牌性格；
- 业务风险；
- 使用频率；
- House Style；
- 产品专用 Token 和例外。

新视觉趋势不得以“更新”为理由破坏基础层；旧原则也不得以“永恒”为理由阻止采用经过目标版本验证的新能力。

### 一致而不相同

跨设备保持：

- 用户任务；
- 信息架构；
- 概念模型；
- 导航语义；
- 数据和状态；
- 品牌识别。

针对设备变化：

- 导航形态；
- 信息密度；
- 布局；
- 控件尺寸；
- 输入方式；
- 动效范围；
- 交互距离。

错误做法：

- 把手机界面等比放大到平板；
- 为每个设备重新发明任务结构；
- 用同一输入路径强迫触摸和键鼠用户；
- 只以屏幕尺寸推断设备能力。

### 连续而不阻塞

用户意图变化时：

- 立即响应；
- 从当前显示状态继续；
- 保持方向和速度连续；
- 允许快速反向；
- 不用动画锁住输入；
- 不让动画延迟业务完成。

错误做法：

- 等动画结束才能继续；
- 从旧目标值开始新动画；
- 松手后速度归零；
- 给高频操作加长动画；
- 将弹簧完成回调作为业务唯一时钟。

### 系统能力优先

优先顺序：

1. 系统控件默认行为；
2. ArkUI 原生能力；
3. 语义化项目封装；
4. 业务特定自定义；
5. 自绘和自定义物理。

自定义的举证责任更高，必须证明：

- 系统能力不能满足任务；
- 没有削弱状态和无障碍；
- 性能可接受；
- 多设备和输入已验证；
- 有维护策略。

### 证据可追溯

任何规则或参数都应回答：

- 谁发布；
- 何时核验；
- 适用什么版本；
- 适用什么设备和输入；
- 是强制、建议、默认还是项目偏好；
- 如何观察；
- 如何验证。

## 2. 产品任务先于界面风格

审视顺序：

1. 用户要完成什么；
2. 主任务和次任务；
3. 任务是否可发现；
4. 路径是否清晰；
5. 状态和错误是否可恢复；
6. 再讨论布局、视觉和动效。

不要使用动效掩盖：

- 错误信息架构；
- 过深导航；
- 不明确的主操作；
- 重复确认；
- 无法撤销；
- 过长等待。

## 3. 导航原则

好的导航让用户知道：

- 我在哪里；
- 我来自哪里；
- 我能去哪里；
- 我如何返回；
- 当前选择是什么。

### 平级导航

适用于同等模块。不同设备可以转换表现：

- 手机：底部页签；
- 平板/PC：侧边导航；
- 大屏：顶部或焦点导航；

具体选择取决于任务、输入和窗口，不由设备名机械决定。

### 层级导航

用于父子内容。检查：

- 返回入口；
- 返回后的状态保留；
- 页面标题；
- 进入和退出方向；
- 深层级的效率。

公开规范建议导航层级通常控制在三层以内。超出时应考虑：

- 搜索；
- 层级穿透；
- 面包屑；
- 分栏；
- 一键回到主界面。

### 混合导航

平级和层级可以组合，但不要：

- 在侧边导航内再嵌套等价的底部页签；
- 让两个导航控件争夺同一层级；
- 让选中态与焦点态混淆。

## 4. 反馈原则

每个可操作元素应在适用输入下反馈：

| 状态 | 目的 |
| --- | --- |
| 正常 | 可操作 |
| 禁用 | 当前不能操作 |
| 按下 | 输入已被接收 |
| 焦点 | 键盘/遥控器当前目标 |
| 激活 | 当前选择或已进入 |
| 悬停 | 光标到达和可交互性 |
| 加载 | 系统正在处理 |
| 成功/失败 | 操作结果 |

反馈可以组合：

- 视觉；
- 触觉；
- 声音；
- 文本；
- 无障碍播报。

多模态反馈应保持因果、时间同步和克制。

## 5. 视觉原则

### vp 与 fp

- 布局尺寸使用 `vp`；
- 字体使用 `fp`；
- 大字体下重新检查布局；
- 只有必须严格控制硬件像素的场景才使用 `px`。

### 网格

- 基础以 8vp 规划；
- 小图标和细节可使用 4vp；
- 网格用于建立秩序，不是把所有值强制取整；
- Margin、Gutter、Column 需要结合设备和内容。

### 分层参数

语义 Token 优先于裸值。覆盖：

- 色彩；
- 字体；
- 圆角；
- 间距；
- 阴影；
- 模糊；
- 缩放；
- 动效。

同一语义 Token 可以在不同设备和主题映射为不同具体值。

### 字体

检查：

- 信息层级；
- 中文、英文、数字混排；
- 文本长度；
- 多语言；
- 动态字号；
- 截断与换行；
- 高密度桌面环境；
- 远距大屏可读性。

## 6. 品牌和愉悦

品牌效果可以存在，但必须：

- 不遮挡任务；
- 不影响高频路径；
- 不制造等待；
- 不削弱可访问性；
- 不大面积反复振荡；
- 有适用时刻。

适合：

- 首次体验；
- 成功完成；
- 关键状态；
- 品牌入口；
- 有明确物理语义的手势。

不适合：

- 每次保存；
- 每个列表项；
- 键盘高频输入；
- 严肃错误；
- 需要快速重复操作的工具界面。

## 7. 新项目启动基线

在编码前形成：

```text
Product Context Card
Evergreen Principle Baseline
Platform Version Profile
Project Personality Overlay
Navigation & Adaptation Plan
Async State Model
First Interactive Prototype Plan
Acceptance Matrix
```

这不是要求先写长文，而是阻止 Agent 在没有设备、输入、平台版本和状态模型时生成大量随机页面。

## 8. AI-first 实验原则

不确定时不要猜：

1. 写问题和假设；
2. 建立最小 ArkUI fixture；
3. 在目标设备运行；
4. 快速连续操作和慢放；
5. 记录性能、无障碍和状态结果；
6. 决定是否修改规则。

模型输出是实验候选，不是事实来源。

## 9. 评审问句

- 该方案服务的首要任务是什么？
- 用户能否预测操作结果？
- 跨设备保持了什么，改变了什么？
- 输入方式变化时是否仍可完成任务？
- 是否优先使用系统组件？
- 视觉参数是否有语义？
- 动效去掉后会损害理解吗？
- 快速反向操作是否连续？
- 大字体和屏幕朗读是否可用？
- 是否有真机性能证据？
