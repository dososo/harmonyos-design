# 严格审视标准

## 1. 证据层级

1. 源码文件与行号；
2. 截图区域；
3. 录屏时间点；
4. 真机和性能数据；
5. 可复现人工观察。

没有证据不输出 Blocker。推断要标明。

## 2. 严重度

- `BLOCKER`：关键任务失败、虚假成功、严重不可访问、严重适配或性能失败；
- `MAJOR`：发布前应修复；
- `MINOR`：一致性和精致度；
- `NOTE`：可选或待验证。

## 3. 动效量级

OpenHarmony 公开参考：

- 简单颜色约 100ms；
- 很短动作约 150ms 内；
- 局部约 200ms；
- 短距离约 250ms 内；
- 复杂约 300ms；
- 长距离或全屏约 350ms 内。

它们是参考，不是统一硬阈值。

## 4. ArkUI Gotchas

- `springMotion`、`responsiveSpringMotion` 和 `interpolatingSpring` 的物理时长不由 `duration` 控制；
- Navigation 默认弹簧可因设备不同，不能与业务计时耦合；
- `geometryTransition` 可能需要关闭系统默认转场；
- 出现/消失优先 `transition`；
- 高频位置/大小变化优先图形变换；
- 同参数变化合并到一个 `animateTo`；
- 不用 `setTimeout` 手写逐帧；
- 无障碍 API 和设备范围必须按目标 SDK 核验。

## 5. 异步真实性测试

至少模拟：

- 2–5 秒延迟；
- 失败；
- 超时；
- 用户取消；
- 双击提交；
- 响应乱序；
- 页面离开后返回。

检查成功视觉和播报是否早于真实确认。

## 6. 真机门禁

- 正常速度；
- 2–5 倍慢放；
- 快速连续输入；
- 反向打断；
- 大数据列表；
- 浅/深主题；
- 默认/大字体；
- 目标输入；
- 屏幕朗读；
- 帧率和长帧。

## 7. 稳定原则与版本化平台

每个发现标记：

```yaml
stability: evergreen | versioned | experimental
design_layer: foundation | platform | project
```

检查：

- 当前视觉趋势是否破坏基础原则；
- API/组件默认是否有目标版本证据；
- House Style 是否被写成平台规则；
- 旧原则是否被用来拒绝经过验证的新能力。

## 8. 可交互原型门禁

以下方案只有静态稿时不得直接通过：

- 连续手势；
- 可打断转场；
- 多窗口或折叠连续性；
- 高风险异步确认；
- 新材质叠加与动态对比度；
- 代码生成产品动效。

最低证据是最小可运行 fixture；正式案例还需要目标设备、快速连续操作、慢放、无障碍和性能记录。

## 9. 跨媒介边界

代码生成的非交互产品动效可以审视节奏、曲线、连续性、排版和几何关系，但不能机械套用按下、焦点、取消和可打断输入规则。传统视频剪辑和叙事不属于本 Reviewer。
